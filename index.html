RISC-NPU tests a small architectural claim in real hardware: a neural operation should not always be treated as a peripheral transaction. By adding a MAC opcode to a custom 32-bit RISC ISA and coupling the NPU directly to the CPU register path, the processor replaces a software accumulation loop with one fixed-latency hardware instruction. The final implementation is verified by Questa simulations and a full Quartus compile on Cyclone IV E hardware, verified in simulation and synthesis at approximately 47 MHz Fmax with positive setup and hold slack, zero DSP block usage, and a 12.3x cycle reduction for the N=50 MAC benchmark.

The Architectural Pattern

The project follows a pattern that appears repeatedly in modern processors: when a workload becomes common enough, the most effective optimization is often not a faster bus or a larger software library. It is a new instruction that removes the overhead around the operation itself.

The research value is not that a multiply-accumulate unit is new. The value is that the entire path is measurable from first principles: the instruction encoding, control states, memory image, datapath muxing, NPU latency, testbench output, synthesis utilization, and post-fit timing are all visible in one small system.

The Problem: A CPU Counting to 2500

Without the MAC instruction, the CPU computes 50 x 50 by clearing register A, loading register B with 50, and executing 50 ADD instructions. Each ADD moves through Fetch, Decode, and Execute. The benchmark records this path at 160 cycles.

SW_cycles = 10 + 3N
N=50 gives 10 + 150 = 160 cycles to produce 2500.

The constant term covers reset settling and setup around the loop. The important part is the slope. Every additional unit of N costs another three CPU cycles. At N=100 the software path is 310 cycles. At N=784, the inner accumulation alone becomes thousands of cycles per neuron.

The final program memory image keeps both paths in one benchmark. The testbench first measures the software loop, then runs the MAC path and checks that both produce the same value, 0x000009C4.

Architecture: One Bus, Two Compute Paths

The CPU remains intentionally small: a custom instruction word, a register file centered on A and B, an ALU path for ordinary arithmetic, and a control FSM that sequences Fetch, Decode, Execute, and MAC_WAIT. The NPU is not attached as an external coprocessor. It is selected through the existing data mux, and when DATA_MUX = "11", the NPU output is written directly into register A.

Opcode Mnemonic Operation Cycles Result Path
0000 0000 LDIA A <- imm[23:0] 3 Instruction immediate to register A
0001 0000 LDIB B <- imm[23:0] 3 Instruction immediate to register B
0111 0000 ADD A <- A + B 3 ALU to register A
0010 0000 STA mem[addr] <- A 3 Register A to data memory
1011 0000 MAC A <- ReLU(A x B) 13 wait cycles NPU to register A

The NPU uses a shift-add multiplier, accumulator, and ReLU stage. Quartus maps it without using embedded DSP blocks, which is useful for teaching because the cost of the acceleration is visible in ordinary logic elements rather than hidden inside a hard multiplier primitive.

The final hardware wrapper matters. Compiling the CPU entity alone proved the core, but it was not the complete FPGA image. The final top entity, risc_npu_top, connects the CPU/NPU system to board clocking, reset, program memory initialized from system_memory.mif, and visible result/status outputs. It is the difference between a simulated processor and a hardware-ready project.

The Clock-by-Clock Story

The control unit exposes a 3-bit state signal, outT, which makes the timing behavior easy to inspect in waveforms. The software path repeats Fetch, Decode, Execute for every ADD. The MAC path enters MAC_WAIT once and holds until the NPU asserts done.

CPU state | 001 Fetch | 010 Decode | 100 Execute | 111 MAC_WAIT
001 Fetch 010 Decode 100 Execute 111 MAC_WAIT

The top ribbon fills with 160 software cycles. The bottom ribbon finishes after the MAC instruction sequence, with the 13 amber cells showing the measured MAC_WAIT period. This is the architectural gain in its most literal form: fewer states, fewer instruction fetches, and no loop overhead.

The Argument Scales

The software path grows linearly because the instruction stream grows linearly. The MAC instruction stays fixed because the wait is defined by the hardware pipeline, not by the numeric size of N in the software loop. The scaling testbench runs N=10, 25, 50, and 100 in one pass.

Verification as Architecture

The most useful engineering lessons came from places where simulation, synthesis, and hardware readiness disagreed. Those failures improved the architecture rather than merely fixing the tests.

ALU simulation correctness

The first ADD tests returned zero because the ALU process read component outputs before the component had advanced by a VHDL delta cycle. The result signals from adder32 were missing from the sensitivity list, so the waveform stayed stale. The final ALU process uses complete sensitivity and variable-based result handling so the simulator and intended combinational logic agree.

Control defaults and latch avoidance

The control unit also needed explicit safe defaults. Without assigning every control output in every path, synthesis can infer storage that the architecture never intended. The final controller starts each combinational evaluation from known defaults, then overrides only the signals required by the current state and opcode.

Synchronous memory timing

The top-level integration exposed a timing detail that unit tests could hide: after a MAC completes, synchronous instruction memory needs a cycle for the next instruction word to be valid. The post-MAC bubble makes the fetch boundary explicit and keeps the hardware behavior aligned with the waveform model.

That is the deeper lesson of the project. A hardware accelerator is not only a datapath block. It is the contract between the ISA, the FSM, the memory timing, and the verification model. The MAC instruction became reliable only when all four agreed.
