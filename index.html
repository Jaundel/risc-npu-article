Modern ML accelerators are separate chips tethered to a host CPU. For edge devices, the dominant cost is not the computation — it is the data movement. This project answers a different question: what if the CPU is the NPU, and neural operations are just more instructions? By extending a custom 32-bit RISC ISA with a tightly-coupled multiply-accumulate unit and adding a MAC opcode, a single processor achieves a 12.3× reduction in cycle count for the MAC operation — verified cycle-accurately in hardware simulation and confirmed in Quartus synthesis at ~47 MHz on a Cyclone IV FPGA.

The Pattern Every Architecture Rediscovers

This thesis does not propose a novel principle. It implements a principle that ARM, RISC-V, and Intel have each independently arrived at — and measures it from first principles in hardware. The fastest accelerator is the one that never leaves the core.

The three papers backing this work — mixed-precision ISA extensions for RISC-V, SIMD MAC extensions for CNN inference, and EXTREM-EDGE for IoT ML — all converge on the same measurement: software loops computing MAC take an order of magnitude more cycles than a hardware instruction. This thesis makes that measurement on a custom processor built from scratch.

The Problem: A CPU Counting to 2500

Without the MAC instruction, the CPU computes A × B by loading A=0, loading B=50, and executing 50 consecutive ADD instructions. Each ADD takes 3 clock cycles — one Fetch, one Decode, one Execute. For N=50:

SW_cycles = 10 + 3N
N=50 → 10 + 150 = 160 cycles to compute 50×50=2500

The constant 10 comes from reset settling, LDIA, and LDIB. Every additional operand costs exactly 3 more cycles. Scale to N=100 and you pay 310 cycles. Scale to a 784-neuron MNIST input and you pay 2,362 cycles — per neuron, per layer, per inference.

The instruction stream for N=100 is not an abstraction. It is 102 words in ROM:

Architecture: One Bus, Two Paths

The CPU is a 4-state FSM — Fetch, Decode, Execute, and for MAC instructions a fourth state: MAC_WAIT. The data path is a single 32-bit bus connecting the register file, ALU, memory, and NPU through a 4-to-1 mux. When DATA_MUX = "11", the NPU result lands directly in register A. No DMA. No external bus. Zero data-movement overhead.

Opcode (bits 31:24) Mnemonic Operation Cycles Path
0000 0000 LDIA A ← imm[23:0] 3 Data bus → Reg A
0001 0000 LDIB B ← imm[23:0] 3 Data bus → Reg B
0111 0000 ADD A ← A + B 3 ALU (adder32) → Reg A
0010 0000 STA mem[addr] ← A 3 Reg A → Data mem
1011 0000 MAC ★ A ← ReLU(A × B) 13 NPU (mul→acc→relu) → Reg A

The MAC instruction is the only one that requires a wait state. The NPU pipeline — shift-and-add multiplier, saturating accumulator, combinational ReLU — runs for a fixed 10 cycles, then raises npu_done. The control unit holds in MAC_WAIT until that signal fires, then latches the result.

The Clock-by-Clock Story

Every clock cycle the CPU asserts outT — a 3-bit state signal readable directly in simulation. The ribbon below maps those signals to color. You can count the difference with your eyes.

The top ribbon is the software path: 160 cycles of repeating blue→green→orange for 50 ADD instructions. The bottom ribbon is the hardware path: the same Fetch/Decode sequence, then a single amber MAC_WAIT block, then done. The amber block is one waiting period instead of fifty loops.

The Bug That Taught the Most

During verification, every ADD instruction produced zero. Register A never changed. The ALU appeared to be dead. The root cause took hours to find — and understanding it requires knowing how VHDL simulation actually works.

Root Cause — VHDL Delta-Cycle Sensitivity List Bug · alu.vhd

The ALU uses an internal adder32 component — a separate entity. In VHDL simulation, component instantiations are concurrent statements: they update their output signals one delta cycle after their inputs change. The ALU process ran, read result_add (stale zero), computed output, and exited. adder32 then updated result_add — but the process never re-ran because result_add was not in its sensitivity list.

The broken line:

process (a, b, op)

The fix:

process (a, b, op, result_add, result_sub, cout_add, cout_sub)

Synthesis hardware was never affected — synthesizers correctly infer the combinational logic regardless. Only simulation was broken. This is a known VHDL pitfall: the simulation model and the synthesis model are not the same thing, and an incomplete sensitivity list silently produces wrong waveforms while the synthesized circuit works fine on hardware.

The fix unblocked the entire verification chain. After adding those four signals, the step-by-step approach confirmed: ALU unit test passed → single ADD CPU test passed → full benchmark passed. Cycle counts matched the formula exactly.

The MAC_WAIT count is 13 for every N. The NPU pipeline depth is fixed in hardware — 10 cycles for the multiplier, plus 3 for state transitions — regardless of the operand values. This is the defining property of hardware acceleration: constant-time execution independent of the operand magnitude.
